ok here is what have to be done we will have two id's in the system , company id and device id
the company id is the TIN it's static and not generated by the client. it's generated by the government , the device id however is just a UUID created by the client and putted in the CSR's SERIALNMBER . the TIN lives in the CSR's ORGANIZATIONNAME field . tha validation flow will look something like this :
1- the client generates a token by visting /onboard 
2- the client generates the csr containing the TIN and the deviceID
3- the server when it receives the enroll request it first checks the token the stored tokens 
4- if valid , stores a new device in the db by storing it's info and the company id as a foreign key , the device info must contain the PIH and the ICV
    this is a security loop hole but there is no way around it the only way to fix it is to have strong onboarding validation before issuing the token , the simplest way this can be done is by letting the company owner issuing a new device enrollment request for a specfic device id , so that the server can validate the request and generate the token safely 
5- this approach is faster than fetching the invoices table and sorting to get the last invoice hash , but it introduces a race condition problem if the same device submitted two invoices at the same time . this needs db locks when reading and writing the icv count and PIH
6- keep in mind that each device has its own ICV and PIH .
